# S5 — Self-Planning Notebook

**Tier:** 1 (self-planning agent — benchmark drives the loop; agent manages its own workspace)
**Metric Group:** G2 / G3 mixed
**Exposure Axis:** `t_steps` (per agent interaction, not per scripted session)
**Default Block Length:** 8–20 interactions per "session"

## What it measures

How well an agent that controls its own filesystem can maintain a useful
working memory across many interactions. Unlike Tier-1 scenarios (S1–S4,
S6) where the benchmark dictates what gets written to memory, in S5 the
**agent decides** what files to create, when to summarize, when to delete.

The benchmark only:
1. Streams in tasks (assistant / KB / coding tasks via `S5Generator`)
2. Resets conversation state every `block_length` interactions (forcing
   the agent to rely on its workspace, not chat history)
3. Scores responses by keyword match plus workspace inspection

This was previously called S7 in the codebase; renamed to S5 in v0.2.

## Aging mechanisms exposed

- Planning complexity scaling with workspace size
- Recall precision across competing similar facts
- Interference between near-duplicate notes
- Update propagation (corrections not applied to all relevant files)
- Decision fragility (inconsistency growing with state)
- Planning fatigue (same task degrading over time)

## Example invocation

S5 works with any `AgentAdapter` — Claude Code, ReactFileAdapter, Codex,
or a custom adapter. Example with Claude Code:

```bash
agingbench run \
  --scenario s5_self_planning \
  --sut agingbench/registry/suts/claude_code/claude_code_self_planning.yaml \
  --sessions 12 \
  --card
```

Or with the local React file-tools adapter (requires `meta-llama/Llama-3.1-8B-Instruct` access on Hugging Face):

```bash
agingbench run \
  --scenario s5_self_planning \
  --sut agingbench/registry/suts/experimental/llama31_react_local.yaml \
  --sessions 10 --card
```
